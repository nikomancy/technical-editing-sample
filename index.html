Channels are containers that hold the messages and other chat data for one or more users. The core idea of channels is simple but they provide a large number of customization options and can address complex use cases.

This page will give you a quick introduction to channels,one of the most impactful objects in the Chat Client SDK and direct you towards more detailed documentation addressing common operations involving channels.

Channels and Channel Types

Every channel has a channel type. If you’ve worked with a class-based object oriented program, the relationship between channels and channel types will be familiar to you.

Channels are objects and Channel Type are classes. You define a Channel Type as a general framework for a use case your app will encounter and instantiate channels from it. Then, as your channel sees use, it may need to mutate its settings.

When this happens, you can change the channel level settings of a channel to something different from the settings it inherits from its channel_type. since these mutations only apply to one specific channel instance, you don’t need to worry about changes for one channel affecting others.

All Stream Chat apps include a few common channel types by default that are built to address some of the most common chat application use cases and to help you defer configuration (and the analysis paralysis that comes with it) while prototyping an application:

Default Channel Types

Livestream

Sensible defaults in case you want to build a chat similar to the one on YouTube or Twitch. Authenticated users are able to post messages but unauthenticated users can still see chat thanks to anonymous users.
Messaging

Settings that let you build something similar to facebook messenger or another messaging app. Two or more authenticated users are able to create channels to communicate with each other.
Team

If you want to build your own version of Slack or Lync, start here. Authenticate users can create channels for specific topics that other users can join or be invited into.
Gaming

Unsure what the specific use case is or the distinctive settings it would have compared to messages or livestream.
Commerce

Users can communicate with sales or support personnel without the friction of creating an account thanks to the guest users feature.

See the full channel type docs for more thorough treatment on channel types and to the channel settings docs for a thorough treatment on setting channel level settings.

Channels and Users

Channels store users associated with the channel in the channel.members property. For more information on working with users, see the full docs.

Note
 While working with a channel using a chat client authenticated with a user JWT, remember that all actions performed by that chat client will be limited to the actions that specfic user has permission to perform. For more information, see user permissions.

Channel Capabilities

The channels feature provides a large set of capabilities and the authenticated user may only have permission to use a small set of these features. Channel capabilities lets a chat client compare the permissions for the current user against the actions a channel provides and return a list of actions that the user has permission to take.

For more information see the full Channel Capabilities docs.

Adding users to a channel

A common use case for channels is that they can have users join and leave. Stream Chat provides a set of methods to help with adding and removing members to channels.

See the Updating Channel Members docs for more information.

Channel Lifecycle

Channels tend to follow a typical lifecycle like the following.

  1. The channel is created.

  2. Users are added to the channel.

  3. The channel handles users and messages.

  4. The channel is end-of-lifed (or continues in perpetuity).

Creating a Channel

Channels can be created either in the front end or on the server. There are several ways to create channels and the correct approach depends on your use case.

Creating Channels by ID

This approach creates a channel with a distinct ID assigned by you. If your use case involves creating channels that serve as containers for communities or specific topics of discussion, creating a channel by id is the way to do it.

See the full docs for more information.

Creating Channels from Lists of Users

This approach creates a channel with a list of users bound into it. If you want to create a channel that functions similarly to a group conversation in Slack, Discord, or Skype, this is the method to use.

Creating a channel with this approach that contains only two users is also how you implement direct message (DM) functionality in Stream Chat.

For more info, see the docs on the channel.create method.

Watching a channel

The channel.watch method is a complex method that provides a lot of functionality in a single bundle. One of its capabilities is to create a new channel if you attempt to watch a channel that does not exist.

For more information on this complex feature, see the full docs.

Channel Use and Operation

After a channel has been made, there are a set of resources it can interact with and actions it can perform.

Channels and Messages

Channels are a container for users and messages. As one of the main objects in Stream chat, you should familiarize yourself with messages.

See the {message-overview-link[message overview docs]

Updating a channel

The channel.update() and channel.updatePartial() methods are methods that give you the ability to update the channel data object associated with a channel. This object contains

  • Metadata about the channel’s creator.

  • The team associated with the channel.

  • Whether the channel is using autotranslation.

  • Whether the channel is frozen or disabled.

  • The members of the channel.

  • Setting overrides.

  • Any custom data you set.

See the Channel Update docs for more information.

Channel End of Life

While a channel can exist forever, there are plenty of examples of chat channels that are built to accomplish a concrete outcome or exist for a finite period of time.

  • A support chat where a support agent needs to communicate with a customer until they resolve the case.

  • A chat that will run alongside a livestreamed event and then conclude when the event is done.

  • A chain of direct messages between two users in a dating app may close when one of the users decides it just isn’t gonna work out.

In cases like these, Stream Chat includes a set of methods to end of life a channel. These methods are granular as, depending on whether you need to retain or dispose of data, end-of-life may mean very different things.

Disable the Channel

Disabling a channel is for cases where:

  • Users should not be able to make any further updates to a channel.

  • Users should not be able to access the data that was in the channel.

  • Applications accessing stream chat with server-side permissions should be able to access and

Disabled channels are inaccessible and effectively deleted, but you have the option to reverse disabling them. this can be done.

See the disabling channels docs for more information on this.

Freezing a Channel

Freezing channels is for cases where:

  • Users should not be able to add or edit messages.

  • You still want users to be able to see the contents of the channel.

Refer to the docs on freezing a channel for more details.

Deleting a Channel

Sometimes you need to delete channels. Stream Chat has two options for deleting channels.

Soft Deleting a Channel

The default deletion provided by Stream Chat is a soft deletion. This means:

  • Clients will not be able to access a deleted channel and its messages

  • The deleted channel and messages cannot be restored.

  • The messages remain accessible for export through a server-side client.

In cases where you need to retain records or don’t need to fully destroy data to meet regulatory requirements such as GDPR’s right to erasure, we recommend using soft delete.

See the documentation on deleting channels for detailed examples of soft deletes.

Hard Deleting a Channel

Certain types of data compliance laws like the GDPR’s right to erasure can require truly removing a channel and its messages from your server. In these cases, Stream Chat offers a hard delete setting.

  • Clients will not be able to access a deleted channel and its messages.

  • The deleted channel and messages cannot be restored.

  • The messages will also be deleted from the server.

Warning
 Hard deletes are final. There is no undo or take-backsies when you hard delete. It’s permanent. If you do not absolutely need to hard delete data in your chat app, we don’t recommend doing it.

See the channel deletion docs for more info and examples of hard deletions.

FAQs

  1. What is the difference between a channel watcher and a channel member?

    • Members is a persistent data association connecting a user to a channel.

    • Watchers is a temporary state indicating the user-client has a websocket connection open to the channel.

  2. Does Stream Support 1:1 conversations?

    Yes. You can create conversations that are exclusive to one or more users.

  3. I’ve used the Mute or Hide Channel method, why is the channel still visible to users?

    The channel.mute() and channel.hide() only affect the visibility of a channel for the user authenticated with the chatClient calling the method. To make a channel invisible or hidden from all users, either delete or disable the channel.
